# SPEC.md

個人ブログプロジェクトのスプリント仕様書。スプリント定義とDefinition of Done（DoD）の唯一の事実源。

## 確定済み技術スタック（変更不可）
- Next.js（`output: 'export'` による静的書き出し）
- コンテンツ管理: Markdown/MDX + Velite（Zodスキーマでフロントマターをバリデーション）
- ホスティング: S3 + CloudFront
- CI/CD: GitHub Actions（mainへのpushでビルド→S3同期→CloudFrontキャッシュ無効化）

## コンテンツ構成（確定済み）
- 記事は `content/blog/{YYYY-MM-DD-slug}/index.mdx` に1記事1フォルダで配置し、画像も同フォルダに同梱する
- フロントマター: `title`, `description`, `date`, `tags`, `published`（`published: false` は下書き扱いでビルド時に除外）

## Sprint 0: プロジェクト基盤とMDX表示パイプライン

### 概要
記事を1本書けば `npm run dev` で表示される、ブログの最小骨格を作る。Next.js初期化、Velite導入、MDXレンダリングパイプラインの整備までを行う。タグ一覧・記事一覧ページ・ページネーション・SNS連携など個別機能は対象外（後続スプリントで定義）。

### 機能要件（ユーザーに見える挙動）
- `content/blog/{YYYY-MM-DD-slug}/index.mdx` に置いた記事が、対応するURLでページとして表示される
- 記事ページには見出し（h2/h3等）にアンカーリンクがあり、クリックでその見出しに遷移できる
- 記事内のコードブロックがシンタックスハイライトされた状態で表示される
- 記事に同梱した画像が記事ページ内に表示される
- フロントマターに `published: false` が設定された記事は、`npm run dev` / `npm run build` のどちらでも一覧・個別ページに表示されない
- フロントマターが必須項目（`title`, `description`, `date`, `tags`, `published`）を満たさない記事がある場合、ビルド時にエラーで検知できる
- ページ全体に最低限のベーススタイル（フォント・余白・読みやすい文字幅など）が適用されている

### タスク分解
1. Next.jsプロジェクトの初期化（TypeScript, App Router, Tailwind CSS導入、`output: 'export'` 設定）
2. Velite導入、`content/blog` 配下を対象としたZodスキーマ定義（`title`, `description`, `date`, `tags`, `published` を含む）、`published: false` の記事をビルド出力から除外する設定
3. MDXパイプライン設定: `rehype-pretty-code`（シンタックスハイライト）, `rehype-slug`（見出しID付与）, `rehype-autolink-headings`（見出しアンカーリンク）を組み込む
4. ダミー記事（画像・コードブロック・複数見出しを含む）を1本作成し、表示確認

### Definition of Done
- [ ] `npm run dev` でダミー記事1本が正しく表示される（必須）
- [ ] ダミー記事内の見出しにアンカーリンクが表示され、クリックでその見出しにスクロール/遷移する
- [ ] ダミー記事内のコードブロックがシンタックスハイライトされて表示される
- [ ] ダミー記事に同梱した画像が記事ページ内に表示される
- [ ] `published: false` を設定した記事が一覧・個別ページのどちらにも表示されない
- [ ] フロントマターの必須項目が欠落した記事を置いた状態で `npm run build` を実行するとエラーで失敗する
- [ ] `npm run build` が（不正な記事がない状態で）正常に完了する
- [ ] ページにTailwind CSSによるベーススタイルが適用されている（無装飾のHTMLでない）

## Sprint 1: 記事一覧ページ

### 概要
Sprint 0時点では `app/page.tsx` にダミー記事1本へのリンクのみが表示される最小状態だった。Sprint 1では、複数記事に対応した本来の記事一覧ページに発展させる。一覧の並び順、各記事の表示内容、記事数が多い場合のページネーションを扱う。タグ機能（タグ一覧・タグ別記事一覧）はSprint 2で扱うため、本スプリントのスコープには含めない。

### 機能要件（ユーザーに見える挙動）
- トップページに記事の一覧が表示される（複数記事が存在する状態に対応する）
- 一覧は記事の日付が新しいものから古いものへの順に並ぶ
- 一覧内の各記事には、タイトル・概要（description）・日付が表示される
- 各記事のタイトル（またはカード/項目自体）をクリックすると、その記事の個別ページに遷移する
- 一覧に表示できる記事数が1ページに収まらない場合、複数ページに分割され、ページ間を移動できる（ページネーション）
- ページネーションには現在何ページ目を見ているかが分かる表示があり、前/次のページへ移動する手段がある
- 最初のページ（1ページ目）には「前のページ」に相当する移動手段が無効化または非表示になる、最後のページには「次のページ」に相当する移動手段が無効化または非表示になる
- フロントマターに `published: false` が設定された記事は一覧およびページネーションの対象件数に含まれない（Sprint 0で実装済みのロジックを踏襲し、一覧表示に発展させた後も成立すること）

### タスク分解
1. ダミー記事を、ページネーションが発生する件数まで増やす（日付がそれぞれ異なる複数記事、うち一部は `published: false` を含める）
2. 記事一覧データを日付の新しい順にソートして取得するロジックを実装する
3. トップページ（`app/page.tsx`）を、1件のダミーリンクから複数記事対応の一覧表示に変更する（タイトル・概要・日付を表示）
4. 一覧の件数が1ページの表示上限を超える場合に複数ページへ分割するページネーション機能を実装する（ページ番号付きURLでの遷移、前/次ページ移動、現在ページ表示を含む）
5. `published: false` の記事が一覧・ページネーションの対象件数から除外されることを確認する

### Definition of Done
- [ ] ダミー記事を、1ページの表示上限を超える件数（ページネーションが最低2ページ以上発生する件数）まで用意した状態で `npm run dev` / `npm run build` の両方を確認する
- [ ] トップページの一覧が、各記事の日付に基づき新しい順に並んで表示される
- [ ] 一覧内の各記事に、タイトル・概要（description）・日付が表示されている
- [ ] 一覧内の記事タイトル（またはカード/項目）をクリックすると、対応する記事の個別ページに遷移する
- [ ] 1ページ目を表示した状態で「次のページ」へ移動すると、1ページ目とは異なる記事群が表示され、かつ日付の新しい順という並び順が全ページを通して維持されている
- [ ] 最後のページまで「次のページ」を辿った後、それ以上次のページへ移動できない（ボタンが無効化または非表示になっている）
- [ ] 1ページ目より前のページへは移動できない（「前のページ」に相当する手段が無効化または非表示になっている）
- [ ] ダミー記事の中に `published: false` を設定した記事を含めた状態で、その記事が一覧のどのページにも表示されず、ページネーションの総ページ数・総件数の算出からも除外されている
- [ ] `npm run build` が複数記事・複数ページの状態で正常に完了する

## Sprint 2: タグ機能

### 概要
Sprint 0・Sprint 1では記事の個別表示と日付順の一覧・ページネーションのみを扱った。Sprint 2では、記事に付与された `tags` フロントマターを活用し、タグによる記事の分類・絞り込み・発見を可能にする。記事ページでのタグ表示、タグ別記事一覧ページ（ページネーション含む）、全タグを見渡せるタグ一覧ページを扱う。

### 機能要件（ユーザーに見える挙動）
- 記事の個別ページに、その記事に付与されているタグ（1つまたは複数）が表示される
- 記事が複数のタグを持つ場合、そのすべてのタグが記事ページに表示される
- 記事ページに表示されたタグをクリックすると、そのタグが付与された記事だけを集めた一覧ページに遷移する
- タグ別記事一覧ページでは、該当するタグを持つ記事のみが表示される
- タグ別記事一覧ページの並び順は、Sprint 1の記事一覧と同様に記事の日付が新しいものから古いものへの順である
- タグ別記事一覧ページにおいても、フロントマターに `published: false` が設定された記事は表示されない
- タグ別記事一覧の各記事には、Sprint 1の記事一覧と同様にタイトル・概要（description）・日付が表示される
- タグ別記事一覧の件数が1ページに収まらない場合、Sprint 1と同様にページネーションされ、前/次ページ移動・現在ページ表示・最初/最後のページでの移動手段の無効化（または非表示）が成立する
- すべてのタグを一覧できるタグ一覧ページが存在し、そこから各タグ別記事一覧ページへ遷移できる
- タグ一覧ページには、各タグに対してそのタグが付与された記事の件数（`published: false` の記事を除いた件数）が表示される
- タグ一覧ページに表示される件数や、そもそも一覧に出現するタグは、`published: false` の記事のみが持つタグについては反映されない（その記事を除けば他に同タグの公開記事がない場合、そのタグはタグ一覧に出現しない、または件数0として扱われない）

### タスク分解
1. ダミー記事のタグ付けを見直し、検証に必要な状態を用意する（複数タグを持つ記事、同一タグを複数記事で共有、あるタグ別一覧がページネーションが発生する件数になるよう記事数を調整、あるタグが `published: false` の記事にのみ付与されている状態を含める）
2. 記事の個別ページに、その記事のタグ一覧を表示する機能を実装する（各タグはタグ別記事一覧ページへのリンクになっている）
3. タグ別記事一覧ページを実装する（指定されたタグを持つ公開記事のみを日付の新しい順に取得し、Sprint 1のページネーション機能を流用して表示する）
4. 全タグとその記事数（公開記事のみを対象にカウント）を表示するタグ一覧ページを実装する
5. `published: false` の記事のみが持つタグが、タグ一覧・タグ別記事一覧のいずれにも公開記事として影響を与えないことを確認する

### Definition of Done
- [ ] 複数のタグを持つダミー記事を用意し、その記事の個別ページにすべてのタグが表示されていることを確認する
- [ ] 記事ページに表示されたタグをクリックすると、そのタグのタグ別記事一覧ページに遷移する
- [ ] 同一タグを共有するダミー記事を、1ページの表示上限を超える件数（タグ別一覧でページネーションが最低2ページ以上発生する件数）用意し、そのタグのタグ別記事一覧ページで日付の新しい順に並んで表示されることを確認する
- [ ] タグ別記事一覧ページにおいても、1ページ目で「前のページ」相当の移動手段が無効化または非表示になり、最後のページで「次のページ」相当の移動手段が無効化または非表示になることを確認する
- [ ] タグ別記事一覧の各記事に、タイトル・概要（description）・日付が表示されていることを確認する
- [ ] あるタグを `published: false` の記事にのみ付与した状態を用意し、そのタグがタグ一覧ページに（件数0または存在しない形で）公開記事に影響を与える形で出現しないことを確認する
- [ ] 同一タグが付与された複数記事のうち一部に `published: false` を設定し、そのタグ別記事一覧ページに非公開記事が表示されず、ページネーションの総件数からも除外されていることを確認する
- [ ] タグ一覧ページに、ダミー記事に存在する全タグが表示され、各タグの記事数が実際の公開記事数と一致していることを確認する
- [ ] タグ一覧ページの各タグから対応するタグ別記事一覧ページへ遷移できることを確認する
- [ ] `npm run build` が、複数タグ・タグ別ページネーションが発生するダミー記事の状態で正常に完了する

## Sprint 3: GitHub ActionsによるS3+CloudFrontへの自動デプロイパイプライン

### 概要
Sprint 0〜2でローカル（`npm run dev` / `npm run build`）での確認のみだった状態から、mainブランチへのpushを起点にビルド・本番環境への反映までを自動化する。今回のスコープはデプロイの自動化のみであり、S3バケット・CloudFrontディストリビューション自体の作成は対象外（事前に手動で作成済みのリソースが存在することを前提とする）。認証はGitHub Actions用のOIDC連携によるIAMロールの引き受けで行い、静的なAccess Key/Secret Keyは発行・保存しない。IAMロールの権限は、デプロイ対象のS3バケットとCloudFrontディストリビューションのみに限定する（最小権限）。

### 前提・制約
- S3バケット、CloudFrontディストリビューションは本スプリント開始時点で既に手動作成済みであり、本スプリントではこれらのリソース自体の作成・設定変更は行わない
- AWS認証にはOIDC連携（GitHubのOIDCプロバイダを信頼するIAMロールをGitHub Actionsが引き受ける方式）を用いる。IAMユーザーの静的なAccess Key ID / Secret Access Keyは発行せず、GitHub Secretsにも保存しない
- IAMロールにアタッチする権限ポリシーは、対象のS3バケット（同期に必要な操作）と対象のCloudFrontディストリビューション（キャッシュ無効化に必要な操作）のみに対するアクセスを許可し、他のS3バケットや他のCloudFrontディストリビューション、その他のAWSリソースへはアクセスできないようにする
- ワークフローファイルの具体的な記法・ステップ構成・使用するGitHub Action等の実装詳細は本スプリントの仕様の対象外とし、実装時に決定する

### 機能要件（ユーザーに見える挙動）
- mainブランチに変更がpushされると、自動的にビルドが実行される
- ビルドが成功すると、ビルド済みの静的ファイルが対象のS3バケットの内容と同期される（mainの最新状態がS3に反映される）
- S3への同期が完了すると、対象のCloudFrontディストリビューションのキャッシュが自動的に無効化され、CloudFront経由でアクセスしたときに最新のビルド結果が反映される
- mainブランチへのpushからCloudFrontキャッシュ無効化までの一連の処理が、人手の操作を介さず自動的に完結する
- ビルドが失敗した場合、S3への同期やCloudFrontのキャッシュ無効化は実行されない
- ワークフローの実行結果（成功/失敗）がGitHub Actions上で確認できる
- デプロイに使用するAWS認証情報として、静的なAccess Key/Secret KeyがGitHubリポジトリのSecretsやワークフローファイルに含まれていない（OIDC経由でIAMロールを引き受けていることが確認できる）
- デプロイに使用するIAMロールの権限は、対象のS3バケットと対象のCloudFrontディストリビューションのみに限定されており、それ以外のS3バケット・CloudFrontディストリビューション・他のAWSリソースに対しては操作が拒否される

### タスク分解
1. GitHub用のOIDCプロバイダをAWS側に設定し、このリポジトリ・ブランチ（main）からの引き受けのみを許可する信頼関係を持つIAMロールを作成する
2. 作成したIAMロールに、対象のS3バケットへの同期操作（取得・書き込み・一覧取得・削除等、同期に必要な操作）と対象のCloudFrontディストリビューションへのキャッシュ無効化操作のみを許可する権限ポリシーをアタッチする（対象リソースのARNを明示的に指定し、ワイルドカードで他リソースを含めない）
3. GitHub Actionsワークフローを作成し、mainブランチへのpushをトリガーに「ビルド→OIDCでIAMロールを引き受け→S3同期→CloudFrontキャッシュ無効化」の流れを自動実行する
4. ビルド失敗時に後続のS3同期・CloudFrontキャッシュ無効化が実行されないことを確認する
5. IAMロールの権限が対象のS3バケット・CloudFrontディストリビューション以外にアクセスできないことを検証する（対象外のS3バケットへの書き込みや、対象外のCloudFrontディストリビューションへのキャッシュ無効化が拒否されることを確認する）

### Definition of Done
- [ ] mainブランチに変更をpushすると、GitHub Actions上でワークフローが自動的に起動する
- [ ] ワークフロー内でビルドが実行され、ビルドが成功した場合に対象のS3バケットの内容がmainの最新ビルド結果と一致する状態に同期される
- [ ] S3への同期後、対象のCloudFrontディストリビューションに対してキャッシュ無効化が実行され、CloudFront経由のURLにアクセスすると最新のビルド結果が表示される
- [ ] mainへのpushからCloudFrontキャッシュ無効化完了までの間、人手による操作（コンソール操作やコマンド手動実行）が発生していない
- [ ] ビルドが失敗するよう変更を加えてpushした場合、S3同期・CloudFrontキャッシュ無効化のステップが実行されず、ワークフロー全体が失敗として記録される
- [ ] ワークフローの実行履歴（成功・失敗を含む）がGitHub Actionsの実行結果画面で確認できる
- [ ] リポジトリのSecrets一覧およびワークフローファイルの内容を確認し、静的なAWS Access Key ID / Secret Access Keyが一切含まれていないことを確認する
- [ ] ワークフローの認証ステップがOIDCトークンを用いてIAMロールを引き受けていることを確認する（IAMロールの信頼ポリシーがGitHubのOIDCプロバイダおよび本リポジトリ・mainブランチに限定されていることを含む）
- [ ] IAMロールにアタッチされた権限ポリシーを確認し、許可対象のリソースが対象のS3バケットと対象のCloudFrontディストリビューションのARNのみに限定されている（`Resource` がワイルドカード `*` のみになっていない）ことを確認する
- [ ] 検証用に用意した対象外のS3バケットへ同ロールで書き込み・削除を試行するとアクセス拒否される（権限エラーになる）ことを確認する
- [ ] 検証用に用意した対象外のCloudFrontディストリビューションへ同ロールでキャッシュ無効化を試行するとアクセス拒否される（権限エラーになる）ことを確認する

## Sprint 4: 共通ヘッダーのバナー画像化と記事一覧へのナビゲーション

### 概要
現状のヘッダーはテキストリンクのみ。Sprint 4では全ページ共通のヘッダーにバナー画像を導入し、どのページからでも記事一覧（トップ = journal/ = `/`）に戻れる導線を明確にする。フッター・検索・サムネイルは別スプリントで扱うため、本スプリントのスコープには含めない。

### 前提・制約
- ヘッダーに使用するバナー画像はプロジェクトの `photo/` フォルダに用意済み（`header.jpg`）。新たな画像素材の作成は本スプリントの対象外。
- 画像配信は `output: 'export'`（静的書き出し）環境で成立する方法を用いる（Next.jsの画像最適化APIに依存しない）。

### 機能要件（ユーザーに見える挙動）
- 全ページ共通のヘッダーにバナー画像が表示される
- バナー画像（またはヘッダー内の明示的なリンク）をクリックすると、記事一覧のトップページ（journal/ = `/`）に遷移する
- ヘッダーは記事一覧・記事詳細・タグ一覧・タグ別一覧など、サイト内のすべてのページで一貫して表示される
- バナー画像には内容が分かる代替テキスト（alt）が設定されている
- ヘッダーのバナー画像が、スマートフォン幅・PC幅のいずれでもレイアウト崩れなく表示される

### タスク分解
1. `photo/header.jpg` を、静的書き出しで配信できる形でプロジェクトに取り込む
2. 全ページ共通レイアウトのヘッダーを、バナー画像を表示する形に変更する（記事一覧トップへのリンクを兼ねる）
3. 代替テキスト・レスポンシブ表示を整える

## Sprint 5: 記事のタイトル・本文による検索

### 概要

### 前提・制約
- ホスティングは静的書き出し（`output: 'export'`、S3 + CloudFront）であり、サーバーサイドの検索APIは使用しない。検索はクライアント側で完結する。
- 検索対象は `published: true` の公開記事のみ（`published: false` の下書きは検索結果に出さない）。

### 機能要件（ユーザーに見える挙動）

- 検索はタイトルと本文の両方を対象とし、本文（記事ページの内容）にのみキーワードが含まれる記事も結果に表示される
- 検索は全公開記事を対象とする（特定のページ送りの1ページ内だけでなく、全記事から一致するものを集める）
- 検索ボックスを空にすると、絞り込みが解除され元の記事一覧（日付の新しい順）に戻る
- 一致する記事が1件も無い場合、その旨が分かる表示（「該当する記事がありません」等）が出る
- `published: false` の記事は、本文・タイトルにキーワードが含まれていても検索結果に表示されない

### タスク分解
1. 公開記事のタイトル・本文テキストを、クライアント側で検索可能な形で用意する
2. 記事一覧ページのヘッダー直下に検索ボックスを設置する
3. 入力キーワードでタイトル・本文を対象に絞り込み、一致した記事を日付の新しい順で一覧表示する処理を実装する
4. 0件時の表示、入力クリア時の復帰挙動を実装する
5. `published: false` の記事が検索結果に出ないことを確認する

### Definition of Done
- [ ] 記事一覧ページのヘッダー直下に検索ボックスが表示される
- [ ] タイトルに含まれるキーワードで検索すると、該当記事だけに絞り込まれる

## Sprint 11: 記事diff検出 + MDX→Markdown変換の共通基盤

### 概要

### 前提・制約
- 記事はMDX形式（Velite管理）であり、Mermaid図・YouTubeEmbed等の独自コンポーネントを含む可能性がある。QiitaおよびZennはプレーンMarkdownのみを受け付けるため、変換処理は必須。
- 画像パスは記事フォルダへの相対パス（`./image.png`）で書かれているが、クロスポスト先は外部サービスのため、CloudFrontドメインを含む絶対URLへの変換が必要。
- 変換スクリプトはTypeScript（Node.js）で実装し、`scripts/crosspost/` に配置する。Veliteのビルドパイプラインとは独立したスタンドアロンスクリプトとして扱う。

### 機能要件（実装詳細）
- mainブランチへのpushで「変更された記事」「新規追加された記事」を `content/blog/*/index.mdx` パターンで検出できる（削除・`published: false` の記事はクロスポスト対象外）
- 変換後のMarkdownから、Veliteのfrontmatterブロック（`---`〜`---`）が除去されている
- MDX独自コンポーネントタグ（`<ComponentName .../>` 形式）が変換時に除去される（コンテンツが失われる場合は空行に置換）
- 記事内の画像パス（`./image.png` 等の相対パス）がCloudFront絶対URLへ変換される
- Qiita向けとZenn向けで、それぞれ適切なfrontmatterを生成できる

### frontmatterマッピング仕様

**Qiita向け（API payloadとして使用）:**

| 元フィールド | Qiita APIフィールド | 変換ルール |
|---|---|---|
| `title` | `title` | そのまま |
| `tags` | `tags` | `[{ name: "タグ名" }]` 形式に変換 |
| `published` | `private` | `!published`（公開=private:false） |

**Zenn向け（Markdownファイル先頭に出力）:**

| 元フィールド | Zennフィールド | 変換ルール |
|---|---|---|
| `title` | `title` | そのまま |
| `tags` | `topics` | string[]形式（最大5件） |
| `published` | `published` | そのまま |
| （なし） | `emoji` | デフォルト値（→ 要決定事項） |
| （なし） | `type` | デフォルト値（→ 要決定事項） |

### 要決定事項
- `SITE_BASE_URL`（画像絶対URL・転載リンク生成に使用するCloudFrontドメイン）: プレースホルダ `https://YOUR_CLOUDFRONT_DOMAIN`
- Zenn frontmatterの `emoji` のデフォルト値（例: `"📝"`）
- Zenn frontmatterの `type` のデフォルト値（例: `"tech"`）
- MDXカスタムコンポーネントの変換方針（単純除去か、代替テキストを挿入するか）

### タスク分解
1. `scripts/crosspost/` ディレクトリを作成し、TypeScript実行環境（tsconfig・必要パッケージ）を整備する
2. diff検出スクリプト（`detect-changes.ts`）を実装する（`git diff --name-only` で変更ファイルを取得し、`content/blog/*/index.mdx` パターンで絞り込む）
3. MDX→Markdown変換スクリプト（`convert-mdx.ts`）を実装する（frontmatter除去・MDXコンポーネント除去・画像パス絶対URL変換）
4. Qiita/Zenn向けfrontmatter生成関数を実装する（マッピング仕様に従い変換）
5. サンプル記事を使って変換スクリプトをローカルで実行し、出力内容（コンポーネント除去・画像URL変換・frontmatterマッピング）を目視確認する

### Definition of Done
- [ ] `scripts/crosspost/` にTypeScriptスクリプトが配置され、ローカルで実行できる
- [ ] diff検出スクリプトを実行すると、変更のある `content/blog/*/index.mdx` のパス一覧のみが出力される（変更していない記事は含まれない）
- [ ] 独自MDXコンポーネント（`<YouTubeEmbed />` 等）を含む記事を変換スクリプトに通すと、コンポーネントタグが除去されたMarkdownが出力される
- [ ] 変換後のMarkdownに `---` frontmatterブロックが含まれない（Veliteのfrontmatterが除去されている）
- [ ] `![alt](./image.png)` 形式の画像パスが、CloudFrontドメインを含む絶対URLに変換されている
- [ ] Qiita向けfrontmatter生成で `tags: ["TypeScript"]` が `[{ name: "TypeScript" }]` 形式に変換されている
- [ ] Zenn向けfrontmatter生成で `emoji`, `type`, `topics`, `published` が含まれている
- [ ] `npm run build` が正常に完了する（スクリプト追加による影響がないこと）

## Sprint 12: Qiita自動投稿（POST/PATCH・ID書き戻し・転載リンク付与）

### 概要
Sprint 11の変換基盤を使い、Qiita API v2を呼び出すGitHub Actionsジョブを追加する。記事の初回投稿はPOST、以後の変更は同記事のPATCH（更新）として扱う。投稿後のQiita記事IDをMDXのfrontmatterに書き戻し、次回以降の更新を冪等に処理できるようにする。

### 前提・制約
- Sprint 11のdiff検出・変換スクリプトが完成していることを前提とする。
- Qiita APIへの認証はQiitaアクセストークンで行い、GitHub Secretsに保存する。
- ID書き戻しは、GitHub ActionsワークフローからMDXファイルをgit commit & pushして行う（`GITHUB_TOKEN` を使用）。
- 投稿済みかどうかの判定はfrontmatterの `qiita_id` フィールドの有無で行う（あればPATCH、なければPOST）。

### 機能要件（実装詳細）
- mainへのpushで `published: true` かつ変更のあった記事がQiitaへ自動投稿または更新される
- 本文末尾に「この記事は筆者のブログからの転載です → [元記事URL]」が自動付与される（元記事URLはサイトのベースURL + 記事スラッグから生成）
- 初回投稿（`qiita_id` なし）: Qiita API `POST /api/v2/items` を呼び出し、レスポンスの `id` を元のMDXファイルのfrontmatterに `qiita_id` として追記してコミットする
- 更新（`qiita_id` あり）: Qiita API `PATCH /api/v2/items/{qiita_id}` を呼び出す
- `published: false` の記事は変更があっても対象外（投稿しない）

### 要決定事項
- Qiitaアクセストークンを格納するSecret名: `QIITA_ACCESS_TOKEN`（仮）
- ID書き戻しコミットのメッセージ形式（例: `chore: write back Qiita IDs [skip ci]`）
- `published: false` に変更された既存投稿記事をQiita上で非公開に変更するか否か（本スプリントでは「対象外（そのまま放置）」とし、必要なら後続スプリントで対応）

### タスク分解
1. VeliteのZodスキーマに `qiita_id`（string, optional）フィールドを追加し、既存記事でも `npm run build` が通ることを確認する
2. Qiita投稿スクリプト（`scripts/crosspost/post-qiita.ts`）を実装する（Sprint 11の変換結果を受け取り、POST/PATCHのAPI呼び出しを行い、レスポンスのIDを返す）
3. ID書き戻しスクリプトを実装する（frontmatterの `qiita_id` を追記してファイルを上書きし、git commit & pushする）
4. GitHub Actionsワークフローに `qiita-crosspost` ジョブを追加する（diff検出 → 変換 → 投稿 → ID書き戻し）
5. `QIITA_ACCESS_TOKEN` をGitHub Secretsに登録する（実際のトークンは運営者が設定）
6. テスト用記事でワークフローを実行し、Qiitaへの投稿・frontmatterへのID書き戻しを確認する

### Definition of Done
- [ ] VeliteのZodスキーマに `qiita_id`（optional）が追加されており、`qiita_id` なしの既存記事でも `npm run build` が正常に完了する
- [ ] mainにpushすると、変更のあった `published: true` の記事がQiitaに投稿または更新される
- [ ] 初回投稿後、元のMDXファイルのfrontmatterに `qiita_id: <Qiita記事のID>` が追記されてリポジトリにコミットされている
- [ ] 同一記事を再度pushしたとき（`qiita_id` あり）、Qiita上で新規記事が増えず、既存記事が更新される（二重投稿しない）
- [ ] Qiita上の記事の本文末尾に「この記事は筆者のブログからの転載です → [元記事URL]」が含まれている
- [ ] `published: false` の記事に変更を加えてpushしても、Qiitaに投稿・更新されない
- [ ] GitHub ActionsのSecretsに `QIITA_ACCESS_TOKEN` が設定されており、ワークフローのログにトークンが平文出力されていない
- [ ] `npm run build` が正常に完了する

## Sprint 13: Zenn連携リポジトリのセットアップ + 自動push実装

### 概要
ZennはAPIを持たないため、Zennが読み取る専用GitHubリポジトリへpushする方式（Zenn公式のGitHub連携）を採用する。Zenn連携リポジトリを作成してZennのGitHub連携設定を行い、auto-blogのGitHub Actionsから変換済みMarkdownをそのリポジトリへ自動pushするジョブを追加する。

### 前提・制約
- Sprint 11のdiff検出・変換スクリプトが完成していることを前提とする。
- Zenn連携リポジトリへの書き込みには、auto-blogリポジトリのデフォルト `GITHUB_TOKEN`（スコープが当該リポジトリのみ）では不足するため、追加の認証情報が必要。

### 機能要件（実装詳細）
- mainへのpushで `published: true` かつ変更のあった記事がZenn連携リポジトリの `articles/{slug}.md` に反映される
- 変換済みMarkdown（Zenn向けfrontmatter付き）が書き出される
- スラッグはMDXフォルダ名（`YYYY-MM-DD-slug`）をそのまま使う（→ 要決定事項）
- 既存ファイルがある場合は上書き（gitの差分pushのためZennで重複記事は生成されない）
- `published: false` の記事は連携リポジトリに反映されない

### 要決定事項
- Zenn連携リポジトリ名: `{GitHub username}/zenn-articles`（仮、確定後に更新）
- 連携リポジトリへの書き込み認証方法: GitHub Personal Access Token（PAT）またはGitHub App（確定後に更新）
- 書き込み認証を格納するSecret名: `ZENN_REPO_PAT`（仮、PAT使用の場合）
- 記事slugの生成規則: MDXフォルダ名（`YYYY-MM-DD-slug`）をそのまま使うか、日付を除いた `slug` 部分のみにするか
- Zenn frontmatterの `emoji` と `type` の決め方（全記事共通デフォルト値か、frontmatterで個別指定できるようにするか）

### タスク分解
1. Zenn連携リポジトリ（`zenn-articles`相当）をGitHubに作成し、ZennのアカウントページからGitHub連携を設定する
2. 連携リポジトリへの書き込みに使うPAT（またはGitHub App）を発行し、auto-blogリポジトリのGitHub Secretsに登録する
3. Zenn向けMarkdownを連携リポジトリに書き出すスクリプト（`scripts/crosspost/push-zenn.ts`）を実装する（連携リポジトリをclone → ファイル差し替え → commit & push）
4. GitHub Actionsワークフローに `zenn-crosspost` ジョブを追加する（diff検出 → 変換 → 連携リポジトリへpush）
5. テスト用記事でワークフローを実行し、Zennのダッシュボードに記事が反映されることを確認する

### Definition of Done
- [ ] Zenn連携リポジトリが作成されており、ZennのGitHub連携設定が完了している（Zennのアカウントページで連携リポジトリが認識されていることを確認）
- [ ] mainにpushすると、変更のあった `published: true` の記事がZenn連携リポジトリの `articles/{slug}.md` に反映される
- [ ] 反映後、Zennのダッシュボードで記事が確認できる（新規記事が表示される、または既存記事が更新されている）
- [ ] 書き出されたMarkdownの先頭に `title`, `emoji`, `type`, `topics`, `published` を含むfrontmatterが含まれている
- [ ] `published: false` の記事は連携リポジトリに反映されない
- [ ] 同一記事を再度pushしても、Zennに重複記事が生成されない（同じ `articles/{slug}.md` が上書きされるだけ）
- [ ] 認証に使うSecret（PATまたはGitHub App秘密鍵）がワークフローのログに平文出力されていない
- [ ] `npm run build` が正常に完了する

## Sprint 14: 冪等性の作り込み・エラー通知・E2E検証

### 概要
Sprint 11〜13で実装した各クロスポストジョブの信頼性を高める。二重投稿・取りこぼしのない冪等な動作を4つのE2Eシナリオで検証し、APIエラーや認証失敗時の通知手段を整備する。本スプリントでコードを大きく書き換えることは想定しておらず、主に動作確認・修正・通知設定が中心となる。

### 前提・制約
- Sprint 12（Qiita）・Sprint 13（Zenn）の実装が完成していることを前提とする。
- エラー通知の基本はGitHub Actionsのワークフロー失敗通知（メール）とする。追加の通知チャンネルは要決定事項。

### 機能要件（実装詳細）
- 同じ記事を複数回pushしても、Qiita/Zennに重複記事が生成されない
- APIエラー（4xx/5xx）発生時にワークフローが失敗として記録され、運営者に通知が届く
- diff検出が正確に機能する（変更していない記事がクロスポストされない）
- Qiita投稿ステップが失敗した場合、ID書き戻しの不整合が起きない（ID書き戻しはQiita APIが成功した場合のみ実行）
- Qiitaのレート制限に引っかかった場合、適切なエラーメッセージを残してワークフローが失敗する

### 要決定事項
- 追加エラー通知チャンネル: Slack / メール / LINE Notify など（GitHub Actionsのデフォルト通知で十分かも含め確定後に更新）
- 追加通知が必要な場合のSecret名・Webhook URL等

### タスク分解
1. E2Eシナリオ1（新規投稿）: テスト用記事を `published: true` でpushし、Qiita・Zennの両方に記事が投稿され、frontmatterに `qiita_id` が書き戻されることを確認する
2. E2Eシナリオ2（更新）: 同じ記事の本文を修正してpushし、QiitaがPATCHで更新（記事IDが変わらない）され、Zenn連携リポジトリが上書き（重複生成なし）になることを確認する
3. E2Eシナリオ3（`published: false` への変更）: 投稿済み記事のfrontmatterを `published: false` に変更してpushし、QiitaへのPOST/PATCHおよびZennへのpushが実行されないことを確認する
4. E2Eシナリオ4（対象外のみ変更のpush）: クロスポスト対象の記事を変更せず、他ファイルのみ変更してpushし、QiitaおよびZennへのAPIコールが実行されないことを確認する
5. APIエラー（認証失敗を意図的に再現して確認）発生時のハンドリングをレビューし、エラーメッセージが適切に残ることを確認する
6. 追加エラー通知チャンネルを設定する（要決定事項が確定した場合のみ）

### Definition of Done
- [ ] E2Eシナリオ1: 新規記事のpushでQiita・Zennの両方に記事が投稿され、frontmatterに `qiita_id` が書き戻されている
- [ ] E2Eシナリオ2: 同一記事の更新pushで、Qiita上の記事IDが変わらず内容が更新され、Zennの連携リポジトリのファイルが上書きされている（Zennで重複記事が生成されていない）
- [ ] E2Eシナリオ3: `published: false` に変更した記事のpushで、QiitaへのPOST/PATCHおよびZennへのpushが実行されない
- [ ] E2Eシナリオ4: クロスポスト対象記事を変更せずにpushしたとき、QiitaおよびZennへのAPIコールが実行されない
- [ ] APIエラー発生時に、ワークフローが失敗として記録され、エラー内容がログに残る
- [ ] ワークフロー失敗時にGitHub Actionsのデフォルト通知が届くことを確認する
- [ ] `npm run build` が正常に完了する
