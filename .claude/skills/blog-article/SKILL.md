このプロジェクトのブログ記事（MDX）を書く・直すときは、以下のルールに従う。
コンテンツ構成の確定仕様は `SPEC.md` を、実際の記述例は `_TEMPLATE_1.mdx` を参照する。

## ファイルの置き場所

- 1記事 = 1フォルダ。`content/blog/{YYYY-MM-DD-slug}/index.mdx` に配置する。
- 画像・サムネイルなど記事に同梱する素材は、同じフォルダにそのまま置く（`./cover.jpg` のような相対パスで参照する）。
- `slug` は英数字とハイフンのみ（日本語・スペース・アンダースコアは使わない）。

## frontmatterの必須項目

`velite.config.ts` のZodスキーマでバリデーションされる。欠落や型違反は `npm run build` でエラーになる。

| フィールド | 型 | ルール |
|---|---|---|
| `title` | string（最大99文字） | 記事一覧・記事ページのタイトル |
| `description` | string（最大999文字） | 一覧のタイトル下に出る要約。検索（タイトル・本文）の対象にはならないが、SNSシェア時の `og:description` に使われる |
| `date` | `YYYY-MM-DD` | 一覧はこの日付の降順で並ぶ |
| `tags` | string配列 | 下記「タグ命名規則」に従う |
| `published` | boolean | `false` は下書き。一覧・ビルド出力・検索結果のいずれにも出ない。公開する記事だけ `true` にする |

### frontmatterの任意項目

| フィールド | 型 | ルール |
|---|---|---|
| `thumbnail` | 画像パス（任意） | 記事一覧の右側に表示されるサムネイル兼OGP画像。`./thumbnail.jpg` のように同梱画像を相対パスで指定する。未設定でもビルドは通り、一覧では `no img` のフォールバック表示になる |

`toc`（目次）は本文の見出しから自動生成されるフィールドで、frontmatterに書く項目ではない。

## タグ命名規則

- 英語の小文字、単語区切りはハイフン（例: `nextjs`, `claude-code`）。日本語タグは使わない。
- 新しいタグを作る前に、既存タグに当てはまらないか確認する。意味が重複するタグ（例: `next` と `nextjs`）を増やさない。
- 既存タグの実績（タグ一覧ページが整理された状態を保つため、まずここから選ぶ）:
  `nextjs`, `react`, `typescript`, `css`, `tailwind`, `velite`, `mdx`, `deploy`, `ux`, `blog`, `draft`
- 1記事に付けるタグは1〜3個程度を目安にする（多すぎるとタグ一覧が分散する）。

## 画像の扱い

- 記事フォルダに画像ファイルを置き、本文では `![altテキスト](./ファイル名.jpg)` で参照する。Veliteがビルド時に最適化・ハッシュ付きファイル名で `public/static/` に出力する（手動でのコピー・配置は不要）。
- alt テキストは必ず書く（空にしない）。
- サムネイル用画像は `thumbnail` フィールドで指定する（本文中の画像とは別物。本文に表示したい画像とサムネイルを兼用したい場合は、同じファイルを両方から参照してよい）。
- 画像素材は `photo/blog/` 配下に記事ごとのフォルダを用意して下書きを置いてよいが、**実際に記事で使う際は `content/blog/{記事フォルダ}/` にコピーしてから参照する**（`photo/` 配下を直接参照しない）。
- ファイル名は記事タイトルと同じにしない。英数字・ハイフンのみの短い名前にする（例: `cover.jpg`, `thumbnail.jpg`, `diagram-01.png`）。
  - 日本語名やスペース入りのファイル名は、パス参照やビルド処理で問題が起きやすいため避ける。
  - `#` `?` `&` のような記号は、URL/アンカー（目次の `#見出し名` 等）と衝突しうるため使わない。
  - 1記事1フォルダの構成上、ファイル名は「その記事フォルダ内で分かれば十分」であり、タイトルを反映する必要はない。
- サムネイルはSNSシェア時の `og:image` としても使われるため、内容が記事の趣旨を表す画像を選ぶ。

## 執筆スタイル

- 見出しは `##`（h2）・`###`（h3）を使う。h2/h3は目次（TOC）に自動で拾われるため、見出し構成がそのまま記事の章立てになる。h1（`#`）は使わない（タイトルはfrontmatterの`title`が担う）。
- コードブロックは通常のMarkdownフェンス（` ```言語名 `）を使う。`rehype-pretty-code` が自動でシンタックスハイライトする。
- 見出し構成は内容に応じて自由に組み替えてよい（`_TEMPLATE_1.mdx` の「背景・本題・まとめ」は一例に過ぎない）。技術手順・日記・トラブルシューティング記録など、記事の性質に合わせる。
- `description` は検索結果や一覧でそのまま読まれるため、本文の冒頭を要約するつもりで1〜2文にまとめる。

## 完了条件（公開前チェック）

- [ ] frontmatterの必須項目（`title`, `description`, `date`, `tags`, `published`）がすべて設定されている
- [ ] `npm run dev` で記事が正しく表示される（見出し・画像・コードブロック・目次を含む）
- [ ] 本文中の画像にすべて alt テキストがある
- [ ] タグが既存タグの命名規則に沿っている（不要な重複タグを作っていない）
- [ ] 公開する場合は `published: true` になっている（下書きのまま残す場合は `false` を維持）
- [ ] `npm run build` がエラーなく完了する
- [ ] 公開時は `git add` → `commit` → `push`（push後、GitHub Actionsが自動でビルド・S3/CloudFrontへデプロイする）
